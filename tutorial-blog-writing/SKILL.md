# 教程/博客写作规范

## 概述

本 Skill 用于指导 AI 撰写中文技术教程。核心目标只有一个：**让一个完全零基础的小白，读完这篇文章后能成为这个领域的专家**。

这意味着：
- 每一个概念都要从零讲起，假设读者什么都不懂
- 每一个术语都要解释到外行能听懂的程度
- 每一步操作都要有代码示例和运行结果，让读者可以跟着做
- 循序渐进，从最简单的开始，逐步深入
- 最终目的是读者看完能独立上手、能解决实际问题

同时，文章读起来必须像真人手写的笔记，而不是 AI 生成的教科书。

## 何时使用

**适用场景：**
- 撰写面向零基础读者的中文技术教程
- 需要消除 AI 生成痕迹，追求真人手写笔记风格
- 博客文章、教学文档等长文技术内容创作
- 要求读者能跟着代码一步步操作的实战教程

**不适用场景：**
- 英文技术文章（语言习惯和写作规范完全不同）
- 纯 API 参考文档（无需教程式的解释和代码示例）
- 小说、散文、随笔等非技术类写作
- 口语化聊天、社交媒体短内容
- 学术论文或正式技术报告（格式要求不同）

## 核心模式

以下是本 Skill 最核心的写作模式，每一条都是区分 AI 文章和手写文章的关键。详细规则见后续章节。

| 模式 | 核心规则 | 详细章节 |
|------|---------|---------|
| 代码文字交替 | 每个知识点后紧跟代码 + 运行结果，禁止大段纯文字 | [代码和文字的比例](#代码和文字的比例) |
| 术语首次解释 | 术语第一次出现时必须用「中文全称（英文，缩写）：通俗解释」格式 | [专业术语解释](#专业术语解释) |
| 表格优先 | API、参数、函数列表等可列表的内容必须用表格 | [表格使用](#表格使用) |
| 不写总结 | 讲完最后一个知识点直接结束，不写总结/下一步/客套话 | [文章结构](#文章结构) |
| 注释自然 | 行内注释格式：`code # 注释`，不做对齐排列 | [代码注释格式](#代码注释格式) |
| 运行结果真实 | 代码运行结果必须从实际执行中复制，不能编造 | [代码可运行性验证](#代码可运行性验证) |
| 安装可验证 | 安装步骤每一步配验证命令，分系统说明 | [安装步骤编写规范](#安装步骤编写规范) |
| 预判排错 | 易出错的代码后紧跟常见报错和解决方案 | [错误处理和排错指南](#错误处理和排错指南) |
| 代码渐进 | 同一概念在不同位置逐级增加复杂度，不为难读者 | [代码的渐进复杂度](#代码的渐进复杂度) |
| 多平台适配 | 命令、路径、包管理器明确标注操作系统 | [多平台适配提示](#多平台适配提示) |

## 语气和称呼

- **假设读者完全零基础**。写每句话之前问自己：一个小白看到这句话能懂吗？如果术语没解释、步骤跳过了，小白就跟不上了。宁可多解释一句，不要默认读者"应该知道"。
- **必须用「笔者」自称**。不要用「我们」「你」「大家」。
- 语气：**像在记笔记或写参考手册**，不像在讲课。去掉「首先让我们来了解一下...」「接下来我们将学习...」这类教学腔。
- **可以诚实表达观点的局限**。比如「笔者使用的是 python3.8」「该教程并不适合完全零基础的读者」——这种坦诚是手写文章的特征。
- 语句尽量简短，能用短句不用长句，能分行不堆成一段。

## 禁止事项（AI 味清除清单）

以下行为**严格禁止**：

1. 禁止「在当今这个 xxx 的时代」开头。
2. 禁止「总而言之」「综上所述」「值得注意的是」「需要强调的是」等套话。
3. 禁止「希望这篇文章对你有所帮助」等客套结束语，禁止「总结」「小结」等文末总结段落——讲完最后一个知识点就结束，不要画蛇添足。
4. 禁止大段纯文字——教程必须代码和文字交替穿插。连续超过 3 段纯文字（无代码、无列表、无表格）是不允许的。
5. 禁止空话编造——不知道某个细节宁可少写。
6. 禁止开头长篇大论的发散或抒情——教程以干货为主，开篇 1-3 句话概述后直接进入正题。
7. 禁止使用 `---` 水平分割线分隔章节——章节之间留一个空行自然过渡即可。

## Emoji 使用规范

**适度使用，自然就好。**

- 用量：一篇文章控制在 **2-5 个** emoji，不要太多也不要零。
- 用法：放在句尾或段尾，作为语气的点缀，不要放在标题中。
- 适用：提示（💡）、警告（⚠️）、强调（🔥）、小情绪点缀（🤔）。
- 禁止：连续多段都加 emoji、把 emoji 当列表项符号。

## 比喻和类比

**适度使用，只在真正帮助理解时加。**

- 用量：一篇文章控制在 **1-2 个**比喻。
- 只用在抽象概念需要具象化的时候，用完即走，不要反复堆砌。
- 禁止：文艺腔比喻（"代码海洋中的巨轮"）、大段寓言式比喻。
- 好比喻的标准：精准、短、像真人闲聊时会打的比方。

## 文章结构

### 开头

**文章内容不以 `# 标题` 开头。** 标题体现在文件名中，文章正文直接从概述开始。章节之间不用 `---` 分割线，留一个空行自然过渡。

```markdown
简短概述 → 官方链接/参考链接（如有）→ 直接进入正题
```

```markdown
xxx 是一个 xxx。它的主要特点是 xxx。

相关网址:

- 官方文档：https://xxx.com
- 教程：https://xxx.com

```

### 层级

**文章内容中不要出现 H1 作为文章大标题。** 正文直接从概述段落开始，第一个章节标题用 `#`（一级标题），子章节用 `##`（二级标题），后面以此类推。

```markdown
# 章节标题

## 小节标题

### 小小节标题
```

### 结尾

讲完最后一个知识点就结束。**不要**总结段落，**不要**「下一步学什么」，**不要**客套话。

## 安装步骤编写规范

安装是新手最容易卡住的环节，必须写得零歧义、可验证。以下是安装步骤的编写铁律：

- **分系统说明**：如果工具在 Windows/Mac/Linux 上安装方式不同，分别列出，不能只写一种
- **每一步都要给出验证指令**：装完一个东西立刻告诉读者怎么确认装成功了，例如装完 Python 立刻接 `python --version` 检查
- **包管理器优先**：能用 `pip install`、`npm install`、`brew install` 搞定的，优先给包管理器命令，而不是让读者去官网下载安装包
- **网络问题预案**：国内读者可能遇到 pip/npm 下载慢或失败，给镜像源方案（如 `pip install xxx -i https://pypi.tuna.tsinghua.edu.cn/simple`）
- **安装成功的样子**：用文字或截图展示安装成功后终端应该输出的内容，让读者能对比确认
- **环境变量等坑**：涉及 PATH、环境变量的，明确写出在哪设置、怎么验证设置生效

安装步骤示例模板：

```markdown
## 安装

### Windows

下载安装包：https://xxx.com/download

安装时勾选「Add to PATH」——这一步很重要，不勾的话后面命令会报错。

装完后打开终端（PowerShell 或 CMD），输入：

```shell
python --version
```

看到类似 `Python 3.8.10` 的输出就说明装好了。

### Mac

```shell
brew install python@3.8
```

验证：

```shell
python3 --version
# 输出：Python 3.8.10
```

### 国内网络问题

如果下载很慢，先换国内镜像源：

```shell
pip config set global.index-url https://pypi.tuna.tsinghua.edu.cn/simple
```
```

## 文章节奏控制

教程不是流水账。不同的内容需要不同的节奏：

| 内容类型 | 节奏 | 做法 |
|---------|------|------|
| 核心概念（如闭包、原型链） | **慢** | 先用白话解释为什么需要它，再给最简代码示例，让读者消化 |
| 基础语法（如变量赋值、循环） | **快** | 给代码 + 注释即可，不用长篇解释，零基础读者看代码+注释能懂 |
| 安装/环境配置 | **细** | 每一步配截图或命令输出，不能跳步骤 |
| 实战项目 | **先慢后快** | 第一个功能拆碎了讲，后面的功能只给代码+关键注释 |
| 进阶技巧 | **快** | 给代码+一句话说明，愿意深挖的读者自己研究 |

节奏控制的几个具体规则：

- 每讲完一个重要概念，给一句过渡再进入下一个概念——不能让读者觉得在跳话题
- 连续 5 个以上简单知识点可以合并成一节快速过，避免文章变成 API 列表
- 全文不超过 3 个「难点」（需要读者停下来理解的概念），超过的话拆成多篇文章

## 多平台适配提示

教程中涉及命令、路径、包管理器的内容，必须显式处理平台差异：

- **命令差异**：Windows 用反斜杠 `\`、Mac/Linux 用正斜杠 `/`；Windows 终端命令不同（`dir` vs `ls`），需要分别给出
- **路径写法**：示例中混用 `F:/environment/...` 和 `/usr/local/...` 是手写文章的正常特征，不需要统一——但要明确标注这是什么系统下的路径
- **快捷键差异**：`Ctrl+C` vs `Cmd+C`，首次出现时标注「Mac 用 Cmd 替换 Ctrl」
- **如果只写了一种平台**：在文章开头声明「本文以 Windows/Mac/Linux 为例」，避免其他平台的读者照着做跑不通

示例：

```shell
# Windows (PowerShell)
python hello.py

# Mac / Linux
python3 hello.py
```

## 标题命名规范

文章的标题（即文件名）直接影响传播效果。遵循以下规则：

- 标题长度控制在 **10-30 字**，过长信息密度低，过短不够明确
- 标题格式：`技术主题：一句话描述`，例如 `Python 装饰器：从闭包到实战`
- **不要**标题党（「全网最全」「一篇文章搞懂所有 xxx」），但要准确传达文章价值
- **不要**在标题中堆砌版本号，版本信息放在文章开头的环境说明中
- 文件名即文章标题，无需在正文中再用 H1 重复

## 图片和截图使用规范

技术教程中的配图应服务于内容，而非装饰：

- **截图必须带标注**：用箭头、框线、编号标注关键区域，不能放一张裸截图让读者自己找
- 截图前调整窗口大小，只截取相关内容区域，避免截全屏
- 代码运行结果截图和文字输出**二选一**——能用文字复制的输出优先用文字（便于读者复制验证），只有图形化界面才用截图
- 图片命名：英文小写 + 连字符，如 `pip-install-success.png`
- **必须配 alt 文本**：`![pip 安装成功界面](pip-install-success.png)`，不要留空 alt

## 代码和文字的比例

- **每个概念/知识点后面必须紧跟代码示例**。让读者看完概念就能立马看到代码怎么用。
- 代码块数量应该等于或超过正文段落数量。
- **必须展示运行结果**。读者需要知道这段代码跑起来是什么样子，才能验证自己写对了。没有运行结果的教程等于没写。
- 使用 bullet + 代码块的紧凑模式：

```markdown
- 返回列表的长度

`len(number)`

- 循环打印列表元素

`for i in number:
    print(i)`
```

- 代码运行结果必须展示出来：

```markdown
运行结果

`a =  [1, 2, 3, 4, ['a', 'b'], 5]
b =  [1, 2, 3, 4, ['a', 'b'], 5]`
```

## 代码的渐进复杂度

教程中的代码不要一上来就给最终版本。读者需要看到代码从简单到复杂的「生长过程」：

- **第一次出现某个概念**时，给最简示例——一个变量、一个函数调用，不需要完整的程序结构
- **第二次出现**时，在前面基础上加一个维度——比如加上循环、加上函数封装
- **最后一次出现**时，给出包含错误处理、边界条件的完整代码

示例（以 Python 文件读取为例）：

第一次出现（最简）：
```python
with open("data.txt") as f:
    print(f.read())
```

第二次出现（加循环）：
```python
with open("data.txt") as f:
    for line in f:
        print(line.strip())  # strip() 去掉每行末尾的换行符
```

最后一次出现（完整）：
```python
try:
    with open("data.txt", encoding="utf-8") as f:
        for i, line in enumerate(f, 1):
            print(f"第{i}行: {line.strip()}")
except FileNotFoundError:
    print("文件不存在，请检查路径")
```

每一版代码之间用文字承接：「上面的写法能跑，但一行一行读更省内存。改一下——」

## 行内代码与代码块的选择

这是一个容易被忽略的细节，但严重影响阅读体验：

| 场景 | 使用 | 示例 |
|------|------|------|
| 文件名、模块名、函数名、变量名在正文中被提到 | 行内代码 | `app.py` 中的 `main()` 函数 |
| 终端命令嵌在正文中 | 行内代码 | 在终端执行 `pip install requests` |
| 配置项的值 | 行内代码 | 把 `timeout` 改成 `30` |
| 完整的代码语句（哪怕只有一行） | 代码块 | 见下方 |
| Shell 命令和它的输出 | 代码块 | 见下方 |
| 配置文件内容（JSON/YAML 等） | 代码块 | 见下方 |

一句话判断：**读者需要复制粘贴的东西用代码块，读者只需要知道名字的东西用行内代码。**

## 代码注释格式

**这是区分 AI 生成代码和手写代码的最关键特征之一。**

### 核心规则

行内注释的格式必须是：**代码 + 一个空格 + 注释符号 + 一个空格 + 注释文本**。不要对齐，不要工整排列。

正确的格式（自然）：

```python
from wayw import app as app_module  # noqa: F821
app_module.config._settings_cache = None  # type: ignore
```

```python
number.pop(1) # 删除第一个元素
number.remove(3) # 删除数组中元素为3的元素
```

```python
self.adb = ADB(self.serialno,adb_path="F:/environment/Android_SDK/platform-tools/adb.exe")
self.adb.wait_for_device()
```

```go
var name string = "张三"  // 显式声明
score := 95               // 短声明
```

```shell
ls -la                     # 详细列表含隐藏文件
cd ~                       # 回主目录
```

### 禁止的对齐注释

以下格式是 AI 的明显特征，**严格禁止**：

```python
# ❌ 禁止：工整对齐的注释
name          = "张三"     # 用户姓名
age           = 25          # 用户年龄
email         = "z@x.com"   # 用户邮箱
```

```python
# ❌ 禁止：注释符号前没有空格
x = 5# 定义变量
```

```python
# ❌ 禁止：注释符号后没有空格
x = 5  #定义变量
```

### 注释内容要求

- 注释只解释**关键逻辑**，不解释「这是什么」的基础语法。
- 注释风格随意——可以加感叹号、口语表达、缩写，不要写得像 API 文档。

```python
# ✅ 好注释：解释为什么
number = 5.454245435
round(number,3) # 保留3位小数

# ❌ 坏注释：描述代码本身
# 定义一个名为 x 的变量并赋值为 5
x = 5
```

### 跨语言规则

| 语言 | 注释格式（正确） |
|------|-----------------|
| Python | `code # 注释` |
| JavaScript/Go/Rust/C | `code // 注释` |
| SQL | `code -- 注释` |
| Shell | `code # 注释` |

所有语言统一：**空格 + 注释符号 + 空格**。

## 表格使用

API 列表、参数说明、函数列表、类型对照——**凡是可以列成表的内容必须用表格**。

```markdown
| 函数 | 作用 |
|------|------|
| len(list) | 返回列表元素个数 |
| max(list) | 返回列表元素最大值 |
```

## 引用和出处标注

凡参考外部资料，必须标注：

- `引用：[链接]` — 借鉴了部分内容但做了改写
- `参考：[链接]` — 借鉴了内容思路
- `转载：[链接]` — 直接引用原资料
- `源自：[链接]` — 该部分内容主要来源于此

## 代码仓库链接规范

如果文章有配套的源代码仓库：

- 在文章开头概述段落之后、第一个章节之前给出仓库链接
- 链接格式：`GitHub 仓库：https://github.com/xxx/xxx`，放在引用块中
- README 至少包含：项目简介、环境要求、安装步骤、运行方法
- 仓库中的代码必须**与文章代码完全一致**，包括注释和变量名
- 代码按章节分文件夹存放（`ch01-basics/`、`ch02-advanced/`），方便读者按进度翻阅

## 代码块必须标注语言类型

**所有代码块必须在开头的三个反引号后面紧跟语言标识符**。禁止使用裸的三反引号（无语言标注的代码块）。

正确写法：

```python
# ✅ ```python
import os
print(os.getcwd())
```

```shell
# ✅ ```shell
ls -la
```

```sql
# ✅ ```sql
SELECT * FROM users;
```

错误写法：

```
# ❌ 裸 ```  —— 不允许！
ls -la
```

常见语言标识对应：`python`、`javascript`、`typescript`、`go`、`rust`、`java`、`shell`、`bash`、`sql`、`html`、`css`、`json`、`yaml`、`markdown`、`plaintext`（纯文本）。

运行结果展示用 `shell` 或 `plaintext`, 如下为实例输出(以linux系统为例)：

```shell
# 运行结果：
linux:~# Hello, World!
```

## 专业术语解释

**每一个专业术语首次出现时必须解释得清清楚楚。** 假设读者是完全零基础的小白，任何缩写或术语不解释等于没写。

### 解释格式

标准格式：**中文全称（英文原名，缩写） + 冒号 + 一句通俗解释**。

```markdown
RPC（Remote Procedure Call，远程过程调用）：一种技术，让你调用远程计算机上的函数
就像调用本地函数一样，不用关心网络传输的细节。
```

### 解释要"说人话"

不要用术语解释术语，要用小白能懂的比喻或直白语言：

```markdown
# ✅ 好的解释
API（Application Programming Interface，应用程序接口）：程序之间沟通的"约定"。
比如你去餐厅，菜单就是 API——你按菜单点菜，厨房按菜单做菜，你不需要知道厨房怎么运作。

# ❌ 差的解释
API（Application Programming Interface）：一组预定义的函数和协议，
用于构建软件应用程序的接口。
```

### 哪些必须解释

| 类别 | 示例 |
|------|------|
| 技术缩写 | API、SDK、ORM、JWT、CSRF、ACID |
| 专业名词 | 容器、闭包、协程、序列化、死锁 |
| 协议名 | HTTP、TCP、WebSocket、gRPC |
| 数学概念 | 矩阵、向量、特征值、奇异值 |
| 工具/框架名首次出现 | Docker、Kubernetes、Nginx、Redis |

### 什么时候不用解释

- 在第一个章节标题（`#`）中出现的基本名词（文章就是讲它的，全文都会解释）
- 已经在前文解释过的术语（第二次出现时不用再解释）
- 极度通用的词（如"文件""文件夹""代码""函数"）

### 解释的位置

在术语**第一次出现的那个句子后面**紧跟解释，用圆括号或破折号引出：

```markdown
Docker 用容器（Container）来运行应用——容器就是一个轻量级的隔离环境，
里面包含了应用所需的全部东西。

Kubernetes（简称 K8s）是一个容器编排平台。
```

## 不同文章类型的适配

所有类型都遵循同样的核心原则：**从小白能看懂的第一步开始，到最终能独立上手解决实际问题**。

| 类型 | 从零到专家的路线 |
|------|-----------------|
| 编程语言 | 安装 → 第一个程序 → 基础语法（变量/类型/控制流）→ 核心特性 → 实战项目 |
| 第三方库 | 这个库解决什么问题 → 安装 → 核心 API 表格 → 每个 API 配代码 → 完整示例 |
| 数据库 | 数据库是什么 → 安装 → CRUD 逐条演示 → 进阶查询 → 用代码语言连接 |
| 命令类 | 工具是什么 → 安装/进入 → 常用命令表格 → 按场景分类示例 |
| 工具类 | 工具解决什么痛点 → 安装 → 核心概念 → 常用操作逐一演示 |
| 理论/数学 | 概念用白话解释 → 公式拆解 → 代码验证 → 实际应用场景 |
| 生活/软技能 | 问题是什么 → 为什么重要 → 具体操作方法 → 实操建议 → 常见误区 |

## 版本标注规范

教程涉及软件版本时，遵循以下规范来避免读者因版本不一致而卡住：

- 文章开头明确标注笔者使用的环境，如：「笔者环境：Python 3.8.10 / Windows 10 / VS Code 1.85」
- 版本号精确到**次版本号**（如 3.8 而非 3.8.10），给读者留弹性空间
- 如果某个特性在特定版本才支持，必须注明：「（需要 Python 3.8+）」
- 教程发布时标注**撰写日期**，方便读者判断内容时效性：「本文撰写于 2024 年 1 月」
- 如果文章内容已过时不适用，在开头用醒目方式标注：「⚠️ 本文内容适用于 xxx 版本，最新版本 API 已变更，仅供参考」

## 代码可运行性验证

教程的核心价值在于读者能**跟着做、跑得通**。如果文章里的代码本身有 bug，读者卡住了就会放弃。写完文章后，必须逐段验证代码能不能跑。

### 验证原则

- **每段可执行的代码都要实际跑一遍**。不能凭感觉说"这段代码应该没问题"。
- 文章中展示的运行结果，必须是从实际运行中**复制粘贴出来的**，不是编造的。
- 如果代码依赖某个环境（特定 Python 版本、安装了哪些库、系统差异），在文章中明确说明。

### 哪些代码必须验证

| 代码类型 | 处理方式 |
|---------|---------|
| 完整可执行的代码片段 | **必须实际运行**，结果与文章一致才算通过 |
| Shell 命令 | **必须实际执行**，确认输出与文章中一致 |
| SQL 语句 | **必须在对应数据库中执行**，确认结果正确 |
| 配置文件（JSON/YAML/TOML 等） | 检查语法正确性、关键字段是否存在 |
| 伪代码或不完整的示意代码 | 标注「示例代码，不可直接运行」 |
| 依赖外部服务的代码（API 调用等） | 至少验证语法正确，尽可能搭建环境验证 |

### 验证流程

1. 准备好文章中描述的环境（版本号、依赖库与文章声明一致）
2. 按文章中出现顺序，逐段拷贝代码执行
3. 对比实际输出与文章中的运行结果
4. 不一致的，修改代码或修改文章中的结果，直到一致
5. 验证通过的，将**实际运行的输出**作为文章中的运行结果展示

### 常见问题处理

- **运行报错**：检查是否遗漏了导入语句、变量定义、前置步骤。教程代码必须是**从上到下可顺序执行**的完整代码。
- **结果不一致**：不同版本、不同系统输出可能略有差异。在运行结果后用括号说明差异范围，或明确指定使用的版本。
- **无法在本机运行**：如果代码需要特殊硬件（GPU）或收费服务，至少用相同语言语法检查工具验证。

## 错误处理和排错指南

好教程和烂教程的分水岭之一：有没有预判读者会遇到的错误。只写「正确代码」的教程，读者跑不通就放弃了。

### 需要预判的错误类型

| 错误类型 | 触发场景 | 教程应该做什么 |
|---------|---------|--------------|
| 环境错误 | 没装依赖、版本不对、PATH 没配 | 给出验证命令和报错信息匹配 |
| 语法错误 | 缩进不对、括号不匹配、拼写错误 | 给出常见的错误提示原文 + 解决方法 |
| 路径错误 | 文件不存在、路径写错 | 教读者用 `pwd` / `ls` 确认当前位置 |
| 权限错误 | 没有读写权限、端口被占用 | 给出 Windows 用管理员、Mac/Linux 用 `sudo` 的提示 |
| 网络错误 | 下载超时、API 不可用 | 给镜像源、离线方案或跳过此步骤的提示 |

### 排错内容的写法

格式：**如果出现 xxx 错误 → 原因是什么 → 怎么解决**

```markdown
如果运行时出现：

```shell
ModuleNotFoundError: No module named 'requests'
```

说明 requests 库还没安装。执行：

```shell
pip install requests
```

如果 pip 下载很慢或报 timeout，换清华镜像源：

```shell
pip install requests -i https://pypi.tuna.tsinghua.edu.cn/simple
```
```

### 排错内容的投放位置

- **紧跟代码块之后**：这段代码最常见的 1-2 个报错，紧接在代码运行结果后面
- **不要在每段代码后都加**：只对容易出错的步骤加排错（安装、网络请求、文件操作、数据库连接）
- **文章末尾可集中放一个「常见问题」小节**：汇总跨章节的通用问题

## 常见错误

以下是撰写技术教程时最容易犯的错误，及其修复方法：

| 错误 | 问题 | 正确做法 |
|------|------|---------|
| 对齐注释 | `name = "张三"     # 姓名` | `name = "张三" # 姓名` |
| 裸代码块 | ` ``` ` 无语言标注 | ` ```python ` |
| 只给代码不给结果 | 读者不知道跑起来什么样 | 每段代码后紧跟实际运行输出 |
| 术语不解释 | 默认读者懂 API、SDK | 首次出现时展开全称 + 通俗解释 |
| 文末写总结 | 「综上所述」「希望有帮助」 | 讲完就结束，不画蛇添足 |
| 大段纯文字 | 连续超过 3 段无代码/表格 | 用代码块或表格打断纯文字段落 |
| H1 做文章标题 | 正文以 `# 标题` 开头 | 标题体现为文件名，正文直接从概述开始 |
| `---` 分割线 | 用水平线隔开章节 | 章节间留一个空行自然过渡 |
| 注释解释语法 | `x = 5  # 定义变量x` | 只注释关键逻辑和"为什么" |
| 教学腔 | 「首先我们来了解一下…」 | 像笔记一样的陈述语气 |
| 安装不给验证 | 装完不告诉读者怎么确认成功 | 每一步附 `xxx --version` 等验证命令 |
| 只写一个平台 | Windows 教程里直接写 `/usr/local/` | 分系统说明，或开头标注适用平台 |
| 一上来给完整代码 | 第一个示例就带异常处理和类型注解 | 从最简版本开始，逐步加复杂度 |
| 只写正确不写报错 | 不给常见错误的排查方法 | 易错步骤后紧跟报错提示和解决方案 |

## 自检清单

生成文章后用以下清单逐条检查：

- [ ] **零基础可读**：一个小白从头读到尾，每一步都能跟着做，不会卡在任何地方
- [ ] 使用了「笔者」自称
- [ ] 标题长度 10-30 字，无标题党，无版本号堆砌
- [ ] emoji 在 2-5 个之间，分布自然
- [ ] 比喻在 0-2 个之间，精准不夸张
- [ ] 安装步骤每一步有验证命令，分系统说明
- [ ] 涉及命令/路径时明确标注操作系统
- [ ] 代码从简单到复杂逐步演进，不是一上来就完整版
- [ ] 易错步骤后紧跟着常见报错和解决方案
- [ ] 行内代码和代码块使用正确（可复制的用代码块，只提名字的用行内代码）
- [ ] 所有代码块都标注了语言类型（```python，不是裸 ```）
- [ ] 代码注释格式正确：`code  # comment`（空格+注释符号+空格）
- [ ] 代码注释没有工整对齐
- [ ] 每个知识点后紧接代码示例，代码有运行结果展示
- [ ] API/参数/函数列表优先用表格
- [ ] 专业术语首次出现时解释得清清楚楚，用说人话的方式
- [ ] 引用外部资料有标注
- [ ] 截图带标注、有 alt 文本，无裸截图
- [ ] 文章开头标注了环境版本和撰写日期
- [ ] 有配套代码仓库的，链接和说明齐全
- [ ] 没有任何「AI 味清除清单」中的禁止行为
- [ ] 文章中没有 `---` 水平分割线
- [ ] 文章内容不以 `# 标题` 开头，H1 仅用作章节标题
- [ ] 讲完最后一个知识点直接结束，无总结、无客套话
- [ ] 文件名即文章标题
- [ ] **代码已验证**：文章中所有可执行的代码都已实际运行过，运行结果与文章一致
- [ ] **最终检验**：读完这篇文章的小白，能不能独立完成这个领域的一个实际任务？
